# 👵👦 3세대 맞춤형 가족 여행 코디네이터 (PoC)

대화로 조건을 알려주면, **계단 유무·아이/안매운 메뉴**는 물론 **오늘 영업 여부**까지 검증해
지도로 추천해 주는 **대화형 식당/숙소 추천 에이전트**입니다.

- **Agent**: LangGraph + LangChain (대화형 상태 관리 그래프 + 후보 검증 루프)
- **LLM**: OpenAI (기본 `gpt-4o-mini`, `agent/config.py` 에서 교체 가능)
- **Backend**: FastAPI
- **Frontend**: Streamlit (상단 지도+결과 / 하단 대화창)
- **Map**: Folium
- **Search**: DuckDuckGo(ddgs) + 네이버 블로그 타겟 검색

---

```
과제/
├── main.py            # FastAPI 서버 (/chat, /chat/stream, /search/step, /search/finish, /reset)
├── app.py             # Streamlit UI (상단 지도+결과 / 하단 대화창)
├── requirements.txt
├── README.md
└── agent/
    ├── __init__.py
    ├── config.py      # .env 로드 + LLM 팩토리(모델명 한 곳에서 관리)
    ├── state.py       # State & 데이터 모델(TripRequirements, Place)
    ├── tools.py       # 웹 검색 / 타겟 리뷰 검색 / 지오코딩 도구
    ├── nodes.py       # ★ State Manager 노드 + 후보 검색/검증/지도 헬퍼
    └── graph.py       # LangGraph 워크플로우 조립
```

## 2. 워크플로우

### (1) 대화형 상태 관리 — LangGraph

```
START → state_manager ──(정보 부족)──► END(되물음)
              └────────(정보 충족)────► END(stage="ready", 검색 준비 완료)
```

| 단계 | 역할 |
|------|------|
| **State Manager** | 대화에서 `누구와/무엇을/어디로` 3단계를 파악해 JSON으로 구조화. 부족하면 되물음, 다 모이면 `stage="ready"` 신호 |

### (2) 후보 검증 루프 — 후보를 '한 개씩' (`/search/step`)

정보가 모두 모이면(`stage="ready"`) 프런트엔드가 `/search/step` 을 **반복 호출**하며,
호출마다 **후보 한 개**를 검색·검증합니다.

1. **후보 검색(리필)** — 후보 대기열이 비면 지역 검색으로 후보 이름 ~10곳을 새로 채움 (이미 본 곳은 제외)
2. **심층 검증** ★ — `"식당명 어린이 메뉴 site:blog.naver.com"` 식 타겟 검색 → LLM이 계단/메뉴 판정
3. **오늘 영업/휴무 확인** — 카카오 장소 상세(`place_url`)의 영업시간을 LLM이 읽어 **오늘 휴무면 제외**(하드 컷), 영업중이면 `🟢 오늘 영업중!` 배지 + 오늘 영업시간 표시 (알 수 없으면 결과에는 올리되 배지 없음)
4. **즉시 추가** — **모든 조건을 통과**한 곳이면 **바로** 최종 목록·지도에 추가

> **최종 5곳**을 채우거나, 후보가 소진되거나, **'그만 찾기'** 버튼을 누를 때까지 반복합니다.
> 매 스텝의 진행 문구(무엇을 조회 중인지)는 별도 스레드 + 큐를 통해 실시간 스트리밍됩니다.
> 지도는 새 후보가 추가될 때만 다시 그리며, 이미 찍은 핀 좌표는 캐시를 재사용합니다.

> 검색을 그래프(단발 invoke)가 아니라 `/search/step` 반복으로 나눈 이유: 결과를 **하나씩 실시간으로** 받아보고, 사용자가 **언제든 멈출 수 있게** 하기 위함입니다.

## 3. 실행 방법

`.env` (실습 루트 `실습/.env`)에 `OPENAI_API_KEY` 가 있어야 합니다.
(선택) `KAKAO_REST_API_KEY` 를 넣으면 지도 핀 정확도가 크게 올라갑니다. 없으면 무료 지오코딩으로 폴백합니다.

```bash
conda activate day4          # langgraph/langchain/ddgs 가 설치된 환경
cd 과제

# 터미널 1 — 백엔드
uvicorn main:app --reload --port 8000

# 터미널 2 — 프런트엔드
streamlit run app.py
```

- 백엔드 문서: http://127.0.0.1:8000/docs
- 프런트엔드: 브라우저가 자동으로 열립니다 (기본 http://localhost:8501)

## 4. 데모 시나리오 예시

1. "부모님이랑 아기 데리고 **서울 송파구**에서 점심 먹을 곳 찾아줘"
2. 에이전트가 부족한 정보를 되물음 → 답변
3. 조건이 모이면 후보를 **하나씩** 검색·검증 → 조건을 모두 통과하고 **오늘 영업 중**인 곳이 **바로바로** 지도·목록에 추가
4. 원하는 만큼 지켜보다가 **'그만 찾기'** 로 중단하거나, 5곳이 채워지면 자동 종료
5. 우측 지도의 핀/목록에서 **통과 조건**(계단/메뉴)과 **오늘 영업시간**을 확인

## 5. 검색 품질 · 지도 정확도

- **상호명·지역 정확도**: 후보 추출 프롬프트가 (1) 메뉴·음식 이름을 상호명으로 착각하지 않도록,
  (2) 검색한 지역에 실제로 있는 곳만 뽑도록 강하게 지시합니다. 지역 근거가 없는 후보(`in_region=False`)는 코드에서 한 번 더 걸러냅니다.
- **지도 핀 정확도**: 지도에는 먼저 **검색 영역을 원(Circle)** 으로 그리고,
  `locate_place` 로 **지역 반경(`SEARCH_RADIUS_KM`) 안에서 확실히 위치를 찾은 곳만 핀**을 찍습니다.
  정확한 좌표를 모르면 **핀을 생략**하고, 목록에는 `🗺️ 위치 미확인` 으로 표시합니다. (엉뚱한 자리 오표시 방지)
  - `KAKAO_REST_API_KEY` 가 있으면 카카오 로컬 검색으로 정확도가 크게 오르고, 없으면 Nominatim 결과가 반경 안일 때만 채택합니다.
- **오늘 영업/휴무**: 카카오 로컬 검색이 주는 `place_url`(장소 상세)에서 영업시간 정보를 가져와
  LLM이 **오늘 요일**의 영업/휴무만 판정합니다. `오늘 휴무 → 최종 결과 제외`(하드 컷),
  `영업중 → 🟢 오늘 영업중! 배지 + 오늘 영업시간`, `알 수 없음 → 결과에는 올리되 배지 없음`.
  (카카오 키가 없으면 `place_url` 이 없어 항상 '알 수 없음' 으로 두어 결과에서 빼지 않습니다.)

## 6. 설계상의 안전장치 (실시간 시연용)

- 모든 도구/함수는 예외를 밖으로 던지지 않고 **폴백**합니다. (검색 실패 → 빈 결과, 지도 중심 지오코딩 실패 → 지역 근사 좌표)
- 후보를 한 개씩 처리하므로 한 곳에서 오류가 나도 다음 후보로 이어집니다.
- 무한 루프 방지: 최대 후보 조사 수(`MAX_CANDIDATES_TOTAL`) 상한을 두었습니다.
- 모델명·검색 개수·검색 반경 등 튜닝값은 `agent/config.py`, `agent/nodes.py` 상단 상수로 분리했습니다.

> 참고: 데모 속도를 위해 기본 모델은 `gpt-4o-mini` 입니다.
> 발표/실사용 시 `agent/config.py` 의 `REASONING_MODEL = "gpt-4o"` 로 바꾸면 판단 품질이 올라갑니다.
