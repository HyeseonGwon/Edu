# Day17_Advanced RAG
좋은 재료와 검증 장치를 갖춰도 여전히 부족한 이유와 해결 방법들

## 1. Query Rewriting
질문 자체가 모호하거나 검색에 적합하지 않은 형태이면 검색에 실패한다.
**핵심 아이디어-Multi-Query**
	LLM이 사용자의 질문을 검색에 최적화된 형태로 재작성 해준다. 여러 개의 다른 표현으로 재작성, 각각 검색한 결과를 통합한다.(앙상블 형태)
**과도한 재작성이 만드는 문제: 질문 Drift 현상**
	원래 질문의 의도에서 벗어난 버전들이 섞여 오히려 무관한 문서를 끌어들이는 경우가 생긴다.
**그래프로 구현하기**
1) 사용자 질문
2) Query Rewrite(Multi-query 생성)
3) 병렬 Retrieve
4) 중복 제거
5) 결과 통합

## 2. HyDE와 Hybrid Search
**HyDE; Hypothetical Document Embeddings "답변으로 답변을 검색한다"** 
	질문 그대로 검색하지 않고 LLM이 먼저 가상의 답변 문단을 생성한 뒤, 그 ==가상 답변을 임베딩해서 검색==한다.
	==개념, 원인을 묻는 질문에는 HyDE가 유리==하지만, 수치, 식별자 등의 엄밀한 결과를 찾아야 하는 문제에는 적합하지 않다. (==그럴듯한 틀린 수치를 지어냄==)

**Hybrid search: 두 가지 검색 방식을 적절히 사용한다**
- 키워드 기반 검색(Sparse); 정확한 단어 일치
- 의미 기반 검색(Dense); 표현이 달라도 뜻이 비슷하면 찾음
- ex) "M3 ETCH, LOT5521 결함 사례" 검색시 "LOT5521"은 키워드 검색, "결함 사례"는 의미 검색으로 각각 찾아야 한다.

**키워드 검색에는 희귀성이 중요하다**
"LOT5521"처럼 특정 문서에만 등장하는 단어는 범용 단어보다 변별력이 훨씬 높다. -> **BM25**는 이 희귀한 단어에 더 높은 가중치를 준다.

**Sparse와 Dense 검색에서 나온 순위는 어떻게 융합하나?**
두 순위표에서 상위에 더 자주 오른 문서에(순위 위치를 점수로 합쳐) 더 높은 최종 점수를 부여한다.

## 3. Contextual Retrieval
"저장 시점에 맥락을 미리 붙여두기"

**Chunking의 한계**
	잘게 나누기 전의 정확한 맥락(인과관계나 배경 정보)을 잃어버린다.
**Context를 유지할 수 있는 전략**은 무엇일까?
	각 chunk의 ==맥락을== LLM이 짧게(50~100단어) ==요약해 붙인 뒤 임베딩==
	맥락 반영 후 키워드 검색을 함께 사용, 정밀 재정렬로 최종 결과 개선
**비용 문제는 어떻게 해결?** 
	원본 문서 전체를 한번 LLM에 전달하여 인덱싱하고, 이후 chunk 요약문들을 Local 캐시로부터 조회한다.

**GraphRAG**(그래프 탐색)
- 문서 속 개체·관계를 그래프로 추출·저장함.
- 질문이 오면 관련 노드·엣지를 따라가며 여러 문서에 흩어진 정보를 연결·추론함.
- 강점: “A와 B가 어떻게 연결되나?”, “원인–부품–알람 관계” 같은 관계형 질문.

**계층 요약 검색** (예: RAPTOR / 트리 요약)  
- 텍스트를 요약해 트리(상위=개괄, 하위=세부) 로 인덱싱함.
- 세부 질문 → 하위 조각, 전체 질문 → 상위 요약을 고름.
- 강점: 질문의 범위(local vs global) 에 맞춰 검색 단위를 바꿈.

> 한 줄 정리: 그래프 = 누구와 무엇이 연결됐나, 트리 요약 = 얼마나 크게/작게 볼까.

## 4. CRAG
사후 처리: 검색 결과를 스스로 고치는 구조
**Retrieval Evaluator**가 검색 결과를 **정확/부정확/애매 3단계로 분류**하여 통째로 쓸지/웹 검색 결과를 함께 쓸지/웹 검색 결과로 대체할지 판단한다.	

**Knowledge Refinement** 문서 안에서도 골라내기
	애매(Ambiguous)한 문서는 작은 단위(Knowledge Strip)로 쪼갠 뒤 실제 관련된 부분만 골라 사용한다.

**CRAG를 그래프로 구현하기**
1) Retrieve Node
2) Retrieval Evaluator Node
3) 조건부 Edge 3갈래(정확/부정확/애매)
	각각 다른 처리 Node로 연결

**Ambiguous 판단이 자주 나올 때의 대응**
기준이 느슨하면 대부분이 애매하다고 판단된다.
구체적인 기준의 예시) "질문의 핵심 키워드가 문서 제목 수준에서 명시적으로 일치하면 Correct로 판단"

## 5. Advanced RAG의 비교
#### Agentic RAG
LLM이 검색 여부, 검색 횟수, 검색 방식 등을 ==스스로 결정하여 단계적으로 수행==

**Agentic RAG가 판단을 하는 방식**
1. 정성(LLM 판단)  
    LLM에게 “추가 검색이 필요한가 / 어떤 방식으로 검색할까”를 직접 물어 단계적으로 결정함.
2. 정량(점수·임계값)  
    검색 유사도·신뢰도 점수를 threshold와 비교해 재검색·중단 등을 결정함.  
    → Adaptive / Agentic에서 흔히 쓰는 일반 기법

**Agentic RAG과 CRAG의 차이** (헷갈리지 말 것)
- CRAG의 핵심은 단순 threshold가 아니라, Retrieval Evaluator가 결과를  
    Correct / Incorrect / Ambiguous 로 분류한 뒤  
    (그대로 사용 / 웹 보완·대체 / Knowledge Refinement)로 분기하는 것임.
- 즉
    - threshold 비교 ≈ “점수만 보고 go / no-go”
    - CRAG ≈ “품질을 3등급으로 분류 후 경로가 갈라짐”

#### Self-RAG란?
각 ==문장 생성 후 근거 여부를 스스로 평가==하여 특수 토큰을 출력하게 한다.

**특수 토큰 Reflection Token이 동작하는 4가지 방식**
	1) 추가 검색 필요성
	2) 질문과 문서의 관련성
	3) 생성한 답변과 문서의 관련성
	4) 답변 전체가 질문에 유용한가?
- 나눠서 단계별로 판단하면 어느 지점에서 근거가 부족했는지 짚어낼 수 있다.

#### Adaptive RAG란?
==질문 난이도를 먼저 판단==해서 간단한 것은 즉시 답변/복잡한 것은 여러 단계에 걸친 검색을 수행한다.

**질문 난이도를 판별하는 법**
	- 질문에 등장하는 개체 갯수
	- 분석을 요구하는 표현(왜, 어떻게 등)
	- 시간 범위 지정 여부
	- 관계 추론 필요 여부
	- 필요한 정보 범위(깊이)

> Adaptive RAG로 먼저 질문 난이도를 나누고, 복잡한 질문에는 Agentic RAG로 검색 전략을 맡기며, 최종 답변에는 Self-RAG처럼 근거를 점검하는 식으로 층층이 쌓아 쓸 수 있음

## 6. RAG 평가 지표와 골든셋(Golden Set)
#### 골든셋이란?
시스템의 성능을 일관되게 측정하기 위한 질문들의 모음. 정답이 정해져 있다.

**골든셋은 어떻게 만드나?**
문항은 최소 20~30문항 이상. 적으면 성능이 좋다고 착각하기 쉽다.
단순 사실 확인, 개념 설명, 관계 추론형 질문 등 여러 유형 및 분야를 섞어야 한다.

**정확도(Precision)**: 검색된 문서==(Top-k 결과)== 중 실제로 질문과 관련 있는 문서의 비율
**재현율(Recall)**: 실제 관련 있는 전체 문서==(정답 집합)== 중에서 검색에 찾아낸 문서의 비율
**근거 충실성(Faithfulness)**: 검색된 문서의 내용과 생성된 답변이 일치하는 정도
	문서에 없는 내용을 추가해서 사실을 왜곡했는지?

**LLM을 평가자로 쓰는 방법**(LLM-as-Judge)
별도의 LLM에게 "검색된 문서와 정답, 생성된 답변을 비교해 점수 매겨줘"한다.
단, 사람이 정기적으로 일부 결과를 직접 검토해서 신뢰도를 관리 및 보정한다.

## 7. 적용 예시
### 1. 질문과 검색 대상을 함께 다듬기
"배선 공정에서 요즘 저항 값이 높게 나오는데 원인 좀 파악해줘" 같은 **구어체** 질문  
→ **Query Rewriting** + **Contextual Retrieval**

### 2. 애매한 결과를 판단하고, 관계까지 추적하기
"열처리 공정 온도 표준값은?" 했는데 다른 공정 문서와 섞여 나오는 상황  
→ **Retrieval Evaluator**가 이를 **Ambiguous**로 분류하면 **Knowledge Refinement**를 거쳐 관련 부분만 추출하고,  
"이 온도 이상이 어떤 부품 문제와 연결되는지"처럼 **관계 추론**이 필요한 질문이 이어지면 **그래프 기반 탐색(GraphRAG)** 으로 넘어감

### 3. 질문 난이도에 따라 다르게 대응하기
"Litho 공정 약자가 뭐야?"처럼 단순한 질문과  
"Litho-15 설비의 최근 3개월 알람 빈도가 늘어난 이유는?"처럼 복합적인 질문이 같은 창구로 들어오는 상황  
→ **난이도 판별(Adaptive RAG)** 단계가 앞의 질문은 **즉시 답변**으로, 뒤의 질문은 **여러 단계 검색**으로 자동 분기시킴

### 4. 개선을 숫자로 확인하기
여러 기법을 추가했는데, 정말 더 나아졌는지 확신할 수 없는 상황  
→ 여러 공정과 여러 질문 유형에 걸쳐 골고루 구성한 **골든셋(Golden Set)** 으로  
이전 버전과 오늘 버전의 **Precision**, **Recall**, **Faithfulness**를 나란히 비교함

9. 질문 난이도를 먼저 보고 단순 질문은 즉시 답, 복잡한 질문은 다단계 검색으로 보내는 방식은 (          ) 임.

### 참(O) vs 거짓(X)

1. Multi-Query는 항상 검색 품질을 올리며 Drift 문제는 거의 없음.  
   → **X** (과도한 재작성 시 **질문 Drift**로 무관 문서가 섞일 수 있음)

2. HyDE는 “결함 코드 LOT5521의 정확한 수치”처럼 식별자·숫자 검증형에 특히 유리함.  
   → **X** (가상 답변이 **틀린 수치를 그럴듯하게 생성**할 수 있어 부적합)

3. `"LOT5521"`은 Sparse(키워드), `"결함 사례"`는 Dense(의미)에 맡기는 식의 Hybrid가 합리적임.  
   → **O**

4. Sparse와 Dense 순위를 융합할 때, 한쪽 목록에만 아주 높이 있어도 다른 쪽 순위는 전혀 반영하지 않음.  
   → **X** (보통 **양 목록의 순위**를 함께 반영해 최종 점수를 만듦. 예: RRF)

5. Chunking만 하면 인과·배경 같은 문맥이 잘려 Contextual Retrieval의 동기가 됨.  
   → **O**

6. Contextual Retrieval의 비용 절감 아이디어는, 문서마다 chunk 요약 시마다 원문 전체를 매번 새로 보내는 것임.  
   → **X** (원문 전체는 **인덱싱 때 한 번** 넘기고, 이후는 **캐시된 chunk 맥락**을 재사용)

7. GraphRAG는 문서 속 개체·관계를 그래프로 두고, 질문을 따라가며 흩어진 정보를 모을 수 있음.  
   → **O**

8. CRAG에서 Incorrect이면 항상 검색 결과만 그대로 써서 답을 고정함.  
   → **X** (부정확하면 **웹 검색으로 대체** 등 다른 경로로 분기)

9. Ambiguous가 너무 자주 나오면 Evaluator 기준을 더 느슨하게 만드는 것이 정석임.  
   → **X** (기준이 느슨하면 더 애매로 감 → **구체적 기준**(예: 핵심 키워드가 제목 수준 일치면 Correct)으로 조임)

10. Agentic RAG는 검색 여부·횟수·방식을 LLM이 단계적으로 스스로 결정하는 편임.  
    → **O**

11. Self-RAG의 Reflection Token을 한 번에만 뭉뚱그려 쓰면, 어느 단계에서 근거가 부족한지 나누어 보기 어려움.  
    → **O** (검색 필요 / 문서-질문 관련 / 답-문서 관련 / 전체 유용성으로 **나눠 판단**하는 게 포인트)

12. Adaptive → (복잡하면) Agentic → (답변 시) Self-RAG 점검은 서로 배타적이라 같이 못 씀.
