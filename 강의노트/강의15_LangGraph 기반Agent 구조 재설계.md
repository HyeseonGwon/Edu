## 1. Prompt에서 Context(맥락) Engineering으로
시스템 프롬프트, 검색된 문서, 대화 이력, Tool의 정의, 장기 메모리... 등 input이 여러 요소일 때는 프롬프트 한 줄로 모두 관리하기 어렵다.

**컨텍스트가 길어질 때 생기는 문제**
	토큰이 많아질수록(==Context Rot== 영역) 모델이 정보를 회상하는 능력은 떨어진다.
	Claude는 의도적으로 최대 토큰을 막아두기도 한다.
#### Context Rot 대응 방법
1) Compaction(압축)
	쌓인 대화 이력과 중간 결과의 ==요약본==으로 새 컨텍스트 시작
2) 구조화된 메모 작성(컨닝페이퍼 만들기)
	작업 중간중간 중요 정보를 파일이나 목록 형태로 컨텍스트 밖에 기록해두고 필요할 때 불러옴
3) **멀티 에이전트 아키텍처**
	복잡한 컨텍스트를 하위 작업 단위로 나누어 여러 Agent가 나누어 담당하게 한다.
	Supervisor 아래에 리서처, 분석, 작성, 검토 Agent가 분업하는 구조. 각 agent들은 특화 업무에 대한 세뇌(system prompt)를 받은 상태이다. 

## 2. Lang Chain에서 Lang Graph로
**Chain**은 Tool을 어떻게 써야하는지 코드에 고정되어 있다.(일직선 파이프라인)
**Graph**는 LLM이 어떤 Tool을 언제, 몇 번 사용할지 스스로 결정한다.(갈림길과 순환 도로)

#### LangGraph의 핵심 구조
- **Node**: 함수 하나가 Node 하나에 대응됨. Node는 Tool이다.
- **Edge**: Node를 연결하는 화살표. 다음에 어느 Node로 이동할지 정의함.
	1) 일반 Edge : 정해진 흐름
	2) ==조건부 Edge = 라우터(Router)== : 조건에 따라 분기하는 결정. 어떤 Tool을 쓸지? 사람에게 넘길지? 어떤 모델을 쓸지? 민감하거나 정책에 위반되는지?
- **State**: 그래프가 공유하는 정보들. Node를 거치며 계속 업데이트된다.
- 결과 평가 Node에서 정보가 충분하지 않다고 판단하면 해당 Node로 다시 돌려보내는 순환 구조를 갖는다. 
	단, 최대 시도 횟수 제한을 해야 무한루프가 방지된다! 토큰 털릴 수 있음

## 3. Router와 ReAct Agent

**ReAct의 패턴** (프롬프트에 충분한 정보가 모이면 답변하라고 했을 때)
-> 생각 "검색이 필요하다"
-> 행동 "검색 Tool 호출"
-> 관찰 "검색 결과 확인"
-> 다시 생각 "추가 검색이 필요하다"

**Agentic RAG**
"충분한가?"를 판단하는 매커니즘
	1) LLM에게 말로 물어보기(정성적)
		하위 질문이 다 커버가 되었는지?
		구체적 수치인지, 일반론인지?
	2) 유사도 점수와 임계값 비교(정량적, CRAG 방식)
		검색에서 나온 유사도 점수가 기준 대비 어떠한지?

**메타 프롬프트**
주어진 프롬프트를 AI가 개선해가면서 여러 가지 버전의 결과를 얻고, 가장 점수가 높은 최적의 프롬프트를 채택한 뒤 ReAct Agent에 적용한다.

## 4. Tool이 많아지면 생기는 문제
- LLM이 혼동하여 잘못된 Tool을 선택할 확률이 증가한다..!
- LLM이 참고해야하는 Tool 스키마가 많아지며 컨텍스트 부담이 가중되기 때문이다.

**해결 방법 : Tool 그룹화**
	모든 Tool을 한 번에 보여주는 것보다, 먼저 어떤 Tool 그룹이 필요할지 판단하게 하면 선택 정확도가 향상된다.
	그룹화는 ==계층적 그래프==로 구현한다.

## 5. MCP: Tool 연결의 표준화
Model Context Protocol : 검색, DB, API 등 다양한 Tool을 표준화된 방식으로 Agent에 연결하는 프로토콜
(특히 오픈 소스에서) 표준 규약을 맞추면 AI 애플리케이션과 외부 도구간 커스텀 연동 코드가 필요 없게 된다.

**MCP 구조: Host, Client, Server**
- 호스트 : 사용자가 쓰고 있는 AI app (Claude desktop, Cursor, ...)
- 클라이언트 : Host 안에 있는 별도 프로세스. 서버와 Client가 1:1 연결.
- 서버 : 실제 기능을 노출하는 독립된 프로그램 (GitHub 서버, DB 서버, ...)

**MCP 서버가 제공하는 3가지: Tools, Resources, Prompts**
- Tools : 모델이 판단해서 호출
- Resource : 앱이 읽어서 제공
- Prompts : 사용자가 선택

## 🛠️ 실제 실행되는 모습
사용자가 **"숫자 맞추기 게임 만들어줘"** 라고 명령을 내리면 시스템은 이렇게 움직입니다.

1. **[하네스 가동]** AI 전용 격리 가상 컴퓨터(샌드박스)가 켜집니다.
2. **[오케스트라 작동]** 중앙 제어 장치가 '기획 에이전트'를 깨웁니다.
3. **[멀티 에이전트]** 기획 AI가 규칙을 짜고, 오케스트라가 이를 전달하자 개발 AI가 파이썬 코드를 작성합니다.
4. **[하네스 통제]** 개발 AI가 짠 코드를 가상 컴퓨터 안에서 실행해 봅니다. 에러가 나면 하네스가 에러 메시지를 가로채서 개발 AI에게 "여기 에러 났으니 고쳐"라고 전달합니다.

**"하네스 엔지니어링**이라는 철저한 규칙 안에서, **오케스트라 AI(회장)** 가 지휘를 하면, **멀티 에이전트(부원들)** 가 각자 맡은 일을 완벽하게 해내서 멋진 게임(결과물)을 완성하는 것입니다!"


- **용어 팁 (Context Rot 관련)**: 요약본에 적힌 'Context Rot'은 대화가 길어질 때 모델의 집중력이 떨어지는 현상을 비유적으로 표현한 것으로 보여. 인공지능 학계나 업계에서는 이를 주로 **'Lost in the Middle'(주요 정보가 중간에 끼어있을 때 모델이 이를 놓치는 현상)** 또는 '컨텍스트 소실/저하(Context Degradation)'라는 표준 용어로 불러. 외부에서 소통할 때는 이 용어들을 혼용하면 더 전문적으로 보일 거야.

### [연습 문제: 빈칸 채우기]

1. 긴 컨텍스트로 인한 모델의 성능 저하를 막기 위해, 복잡한 작업을 하위 단위로 쪼개어 Supervisor와 여러 하위 Agent가 분업하도록 설계하는 방식을 ( **멀티 에이전트** ) 아키텍처라고 한다.
    
2. LangGraph에서 흐름의 갈림길 역할을 하며, 조건이나 LLM의 판단에 따라 다음에 실행할 노드를 동적으로 결정하는 요소를 ( **조건부 Edge** 또는 **라우터** )(이)라고 한다.
    
3. 오픈소스 및 AI 애플리케이션과 외부 도구(DB, API 등) 간의 커스텀 연동 코드 없이 표준화된 규격으로 도구를 연결해 주는 프로토콜의 명칭은 ( **MCP** 또는 **Model Context Protocol** )이다.

### [연습 문제: 참(O) / 거짓(X) 판단]

**1. LangGraph 구조에서는 노드 간의 순환(Loop)이 가능하므로, 무한 루프에 빠져 토큰이 과다 소모되는 것을 방지하기 위해 최대 시도 횟수 제한(recursion_limit)을 반드시 설정해야 한다.**

- **정답: O** (요약본에 언급된 것처럼 그래프 구조의 순환 특성상 제한 장치가 필수적이야.)
    

**2. Tool의 개수가 너무 많아지면 LLM이 참고해야 하는 스키마가 늘어나 컨텍스트 부담이 생기는데, 이를 해결하기 위해 모든 Tool을 하나의 평면적인 그래프 구조에 병렬로 배치해야 한다.**

- **정답: X** (모든 Tool을 한 번에 보여주면 혼동이 오기 때문에, Tool을 그룹화하여 계층적 그래프(Hierarchical Graph)로 구현하는 것이 올바른 해결책이야.)
    

**3. MCP(Model Context Protocol) 서버가 호스트 애플리케이션에 제공하는 세 가지 핵심 기능 요소는 Tools, Resources, Vectors이다.**

- **정답: X** (MCP 서버가 제공하는 3가지는 Tools, Resources, 그리고 사용자가 선택할 수 있는 **Prompts**야.)
