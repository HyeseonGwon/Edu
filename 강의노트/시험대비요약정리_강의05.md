### **1. LangChain 개요 및 프로덕션 한계
 * **개념**: LLM 기반 앱 개발 시 반복되는 코드 패턴을 **추상화** 및 **모듈화**하여 레고 블록처럼 간편하게 조합할 수 있도록 돕는 표준 프레임워크.
 * **모듈화**(Modularity), **조합 가능성**(Composability), **확장성**(Extensibility)을 3대 핵심 가치로 두어 모델이나 DB 교체를 한 줄의 코드 변경으로 용이하게 만듦.
 * **한계**: 빠른 프로토타입 개발에는 유리하나, 프로덕션 환경 적용 시 **레이턴시(응답 지연)** 증가, 기본 설정으로 인한 **비용 폭탄**, 복잡한 추상화 레이어로 인한 **디버깅 지옥**이 발생할 수 있음.
### **2. Model, Message Type 및 Output Parser
 * **개념**: LangChain에서 AI 모델 연동 및 출력 데이터 가공을 담당하는 기본 모듈.
 * **구조**: 메시지 타입은 AI의 가이드라인을 설정하는 **System**, 사용자의 질문인 **Human**, 모델의 답변 기록인 **AI** 메시지로 구조화됨.
 * **Output Parser**: LLM이 반환하는 단순 문자열(텍스트)을 시스템이나 코드에서 다루기 쉬운 리스트(List)나 **JSON** 등의 Python 자료형으로 자동 변환해 줌.
	 * LLM마다 출력 형식이 조금씩 다르다. Output parser는 LLM을 교체해도 동일한 출력 형식을 보장한다. 출력의 사용 목적에 따라 적절한 parser를 선택하면 후속 활용이 더 쉽고 안정적으로 이루어진다.
### **3. LCEL(LangChain Expression Language)과 Chain
 * **개념**: 컴포넌트들을 **| (파이프 연산자)** 로 엮어 컨베이어 벨트처럼 데이터가 **왼쪽에서 오른쪽으로** 흐르게 만드는 선언적 표현 방식.
	 *ex) prompt | llm | output_parser
 * **특징**: 모든 핵심 컴포넌트가 **Runnable** 인터페이스를 공통 구현하고 있어 자유로운 결합이 가능함. 원래 질문 등의 원본 입력을 유실 없이 그대로 보존해 전달하는 **RunnablePassthrough**나 병렬 처리를 위한 **RunnableParallel**을 지원한다. 
 * 복잡한 업무를 단계별로 나눠 처리하는 **멀티 체인** 지원.
 * 실시간 토큰 단위 출력을 위한 **스트리밍(.stream())** 기능을 지원.
### **4. Memory (대화 기억 메커니즘)
 * **개념**: 각 요청이 독립적이며 이전 대화를 기억하지 못하는 LLM의 **Stateless(무상태)** 특성을 극복하기 위해 도입된 컴포넌트.
 * **특징**: 이전 대화 내용을 현재 프롬프트에 자동으로 삽입하여 맥락을 유지시킴. 전체 이력을 저장하는 Buffer Memory는 비용 급증 한계가 있으므로, 실무 다중 사용자 환경에서는 **session_id**를 기준으로 대화 이력을 완전히 독립 분리하여 관리해야 함.
 * **Buffer Memory**: 대화 이력 전체를 요약 없이 그대로 버퍼에 저장. 단순하나 토큰 수 급증
 * **Window Memory**: 최근 N개의 대화만 보존
 * **Summary Memory**: 이전 대화를 요약해서 저장. 긴 대화에 비용 최적화.
### **5. 프로덕션 RAG 실패 원인
 * **개념**: 외부 지식 소스에서 관련 문서를 검색하여 LLM 답변의 신뢰성을 보강하는 파이프라인 구현 방식.
 * **실패 요인**: 실제 프로덕션 환경 실패의 상당수는 **Chunking(문서 분할)** 단계의 잘못된 설계에서 비롯됨. 시스템이 오답을 생성했음에도 아무런 에러 신호 없이 당당하게 오답을 출력하는 **침묵하는 실패(Silent Failure)** 현상이 가장 위험함.
**==RAG + Memory==**: AI가 이전 맥락을 기억하면서 + 문서를 찾아서 + 근거 기반으로 답변한다.
### **6. 이미지 처리 및 최신 데이터 증강
 * **개념**: 컴퓨터가 인지하는 픽셀 값을 조절·가공하거나 학습용 이미지 데이터 부족 한계를 극복하기 위한 기술.
 * **기법**: 모든 픽셀에 일정 값을 더하거나 빼서 밝기를 조절하는 **정규화(Normalization)**, 노이즈 제거 및 특징 추출을 위한 **필터링(Filtering)**, 히스토그램을 넓혀 대비를 향상시키는 **평활화(Equalization)** 가 있음.
 * **k-means 알고리즘으로 이미지 단순화**: k개로 군집화->색상 데이터를 k개로 줄임(cv2.kmeans)
 * **최신 데이터 증강**: 무작위 영역의 픽셀을 제거하고 다른 이미지 패치로 대체하며 레이블도 비율대로 혼합하는 **CutMix**(ex. Dog 0.5+Cat 0.5), 사각형 영역을 무작위로 지우고 난수로 대체하여 모델을 폐색(가려짐) 상태에 **견고(Robust)** 하게 만드는 **Random Erasing** 등이 활용됨.

### [연습 문제: 빈칸 채우기 및 참/거짓]
#### 1. 빈칸 채우기
(1) LangChain의 컴포넌트들이 공통 규격으로 구현하고 있어 파이프 연산자(|)를 통한 연쇄적 결합을 가능하게 만드는 기반 인터페이스는 ( **Runnable** )(이)다.
(2) 이미지 데이터 증강 기법 중, 단순히 픽셀을 검게 지우는 기법을 넘어 지워진 영역을 다른 이미지의 패치로 채우고 레이블도 비율에 맞춰 혼합해 주는 최신 기술은 ( **CutMix** )(이)다.
(3) 여러 사용자가 동시에 장비를 사용하는 실무 환경에서 각 엔지니어의 대화 내용이 서로 섞이지 않도록 독립적으로 관리하기 위한 식별 기준 값은 ( **session_id** )(이)다.
(4) LLM이 과거의 대화 흐름을 스스로 기억하지 못하고 매번 들어오는 입력 요청에 대해서만 완전히 독립적으로 처리하는 한계 특성을 ( **Stateless / 무상태** )(이)라고 한다.
#### 2. 참(O) vs 거짓(X) 판단
(1) LCEL(LangChain Expression Language) 파이프라인 구조 내에서 데이터는 연산자 규칙에 따라 오른쪽에서 왼쪽 방향으로 흐른다. ( **X** )  *(설명: 데이터는 왼쪽에서 오른쪽으로 흐름)*
(2) 대화 이력 전체를 요약 없이 그대로 보관하는 Buffer Memory 방식은 대화가 길어질수록 토큰 수가 급증하여 API 비용 폭탄 및 컨텍스트 창 한계 초과 문제를 야기할 수 있다. ( **O** ) 
(3) 이미지 평활화(Equalization) 기법은 이미지의 히스토그램이 특정 영역에 과도하게 집중되어 대비(Contrast)가 낮아진 상태일 때 이를 넓게 펼쳐 품질을 개선해 준다. ( **O** ) 
(4) LangChain 프레임워크는 파이프라인의 구조적 연결을 편리하게 도울 뿐이므로, 애초에 RAG 파이프라인 설계가 잘못되었거나 프롬프트 구성이 나쁘다면 품질 저하 문제를 자동으로 해결해 주지 못한다. ( **O** )
