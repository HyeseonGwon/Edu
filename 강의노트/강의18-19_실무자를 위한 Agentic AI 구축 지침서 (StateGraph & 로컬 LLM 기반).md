본 지침서는 복잡한 비즈니스 로직과 수일~수주에 걸친 장기 프로세스를 자율적으로 처리하는 **StateGraph 기반 Planning Agent**를 구축하고, 이를 실제 운영 환경에 안전하게 배포·통제하기 위한 실무 가이드라인입니다.
## [지침 1] 단발성 QA를 넘어선 계획 수립: Plan-and-Execute 패턴
### 1. 단발성 질의응답과 장기 작업 계획의 차이
 * **기존 단발성 QA의 한계:** 하나의 질문에 집중하여 검색, 판단 후 근거 있는 정확한 답변을 일회성으로 제공하는 데 머물렀습니다.
 * **장기 작업의 핵심:** "여러 날에 걸친 신제품 공정 검증 계획 수립"과 같이 복잡한 단계를 순서대로 실행하고, 중간에 사람의 승인을 받으며, 문제가 생기면 이전 단계로 되돌아가 계획을 수정 및 재실행하는 장기적인 자율 흐름을 관리해야 합니다.
### 2. ReAct 패턴 vs Plan-and-Execute 패턴
 * **ReAct 패턴:** 매 순간 한 걸음씩 생각하고 판단하여 행동하므로 경로가 지그재그 형태로 꺾이며 복잡한 장기 작업에서는 비효율적인 경로를 생성합니다.
 * **Plan-and-Execute 패턴:** 사용자 요청(예: 예산 500만 원, 인원 20명 팀 워크숍 기획)이 유입되면 먼저 전체 하위 계획을 수립(**Planning**)한 뒤, 계획된 순서대로 하나씩 자율 실행(**Execution**)하고, 제약 조건을 만족하는지 최종 검토(**Review**)하는 구조로 단계를 명확히 분리하여 직선형 경로로 순차 실행합니다. 제약 조건 미달 시 Planning 단계로 복귀합니다.
### 3. 복잡한 계획의 계층화 (Hierarchical Planning)
 * 하위 작업이 10개, 20개 이상으로 늘어날 경우 평평한 목록(Flat List) 구조는 전체 흐름 파악 및 진행 상황 추적이 불가능합니다.
 * 큰 단계(예: 시험 생산 \rightarrow 품질 검증 \rightarrow 양산 이관)를 먼저 나누고, 각 거대 단계 내부에서 세부 작업을 계획하는 **2단 이상의 계층 구조**로 설계해야 전체 흐름 파악과 집중 관리가 용이해집니다.
 * **계획 수립의 핵심 원칙:** 모든 계획은 **"각 단계가 성공했는지 실패했는지 명확히 판단할 수 있는 크기"**로 분할·구성해야 합니다. 너무 세밀하면 사소한 변경에도 전체 재계획이 유발되며, 너무 뭉뚱그리면 실행 중 문제 발생 지점을 추적할 수 없습니다.
## [지침 2] 계획 전체를 기억하는 정교한 State 및 체크포인트 설계
### 1. 정교한 State 딕셔너리 구조 설계
장기 작업은 프로그램이 중단되거나 사람이 개입해도 재시작 위치를 정확히 알아야 하므로 아래 필드를 포함하는 정교한 State 설계가 필수적입니다.
 * plan: 전체 계획 목록과 각 단계의 진행 상태(done, in_progress, pending) 보관.
 * current_step: 현재 실행 중인 단계 번호로, 중단 후 재시작 시 이어갈 위치 식별.
 * step_results: 각 단계에서 생성된 결과 요약 및 산출물(파일, 링크 등) 저장.
 * approval_log: 사람의 승인 이력(누가, 언제, 어떤 결정)을 기록하여 되돌리기 및 반려 판단에 활용.
 * context: 전체 목표, 예산/인원/마감일 같은 제약 조건 및 전반적인 맥락 정보.
 * notes: 중요 메모, 특이사항, 다음 액션 등 사람과의 커뮤니케이션용 자유 메모 공간.
### 2. 영속성 레이어 구축: 체크포인트(Checkpoint)
 * **메모리 저장의 한계:** State를 프로그램 메모리에만 두면 서버가 재시작되거나 오류가 발생하는 순간 지금까지의 계획과 진행 상황이 통째로 소실됩니다.
 * **외부 저장소 활용:** 각 단계가 끝날 때마다 State 전체를 별도의 외부 저장소(PostgreSQL, Redis 등)에 체크포인트로 기록하고, 재시작 시 최신 State를 불러와 안전하게 이어갑니다.
 * **특정 시점 되돌리기(Rollback):** 후속 단계 실행 중 과거의 판단 오류나 이슈를 발견하면, 시스템 상태를 오류 발생 전 단계(예: 3단계 시점)로 되돌려 그 이후 단계만 수정 반영하여 다시 실행할 수 있습니다.
### 3. 재계획(Re-planning) 유발 신호 판단 기준
실행 중 사소한 편차와 전제를 바꾸는 문제를 명확히 구분하여 흐름을 통제해야 효율적인 진행이 가능합니다.
 * **그대로 진행 (사소한 편차):** 예산 500만 원 중 실제 505만 원 지출(\pm 1\%)이나 전체 일정에 영향이 없는 사소한 일정 지연, 대체 데이터로 보완 가능한 누락은 파라미터 조정 및 부분 수정을 거쳐 불필요한 재계획 없이 다음 단계로 계속 진행합니다.
 * **재계획 필요 (전제를 바꾸는 문제):** 예산이 300만 원으로 대폭 축소되거나, 핵심 설비 공급이 불가능해지거나, 새로운 규제 발생 등으로 애초의 전제가 틀어진 경우에는 현재 State를 저장하고 Planning 단계로 복귀하여 계획을 재정렬합니다.
## [지침 3] 자율성과 통제의 균형: 퀄리티 게이트와 HITL
### 1. 퀄리티 게이트(Quality Gate)의 자동화 통제
작업이 다음 단계로 넘어가기 전, 정해진 기준을 통과해야만 진행을 허용하는 검문소로 "확인부터 하고 진행"하도록 순서를 바꾸는 장치입니다. Review 단계를 구체화한 것입니다.
 * **통계적 기준:** 연속형 데이터 분포 기반 처리 방식으로 최근 데이터 평균에서 표준편차의 2배(\pm 2\sigma) 범위를 만족하면 정상 통과, 벗어나면 이상치로 자동 판단하여 보류합니다 (예: 수율, 공정 시간, 온도 등).
 * **업무 규칙 기준:** 명시적 규칙 적용 방식으로 조직의 합의된 기준이나 컴플라이언스를 즉시 적용합니다 (예: 예산 초과율 10% 이하 시 통과 / 초과 시 보류, 일정 지연율 5% 이하 시 통과 / 초과 시 보류).
 * **판정 결과 분기:** 기준 만족 시 **통과(PASS \rightarrow 다음 단계)**, 추가 확인 필요 시 **보류(HOLD \rightarrow 재검토/보완 요청)**, 기준 미달 시 **반려/재작업(FAIL \rightarrow 계획 수정 후 재시도)** 경로를 조건부 엣지로 구현합니다.
### 2. Human-in-the-loop (HITL) 안전판 구축
 * **개념:** 자동화된 기준으로 판단하기 애매하거나 결과의 파급력이 커서 위험 부담이 높은 핵심 순간에 그래프 실행을 일시 중단(Interrupt)하고, 사람에게 확인 및 승인을 요청한 뒤 재개하는 안전 장치입니다.
 * **실무 조합 원칙:** 모든 단계에 HITL을 배치하면 병목이 생겨 효율이 저하되므로, **위험도와 파급력에 비례하여 설계**해야 합니다. 80~~95%의 정형 업무는 퀄리티 게이트로 자동 처리하고, 고액 환불이나 중요 예산안 승인처럼 위험도가 높은 5~~20%의 예외 케이스만 선별하여 HITL 구조를 결합합니다.
## [지침 4] 컨텍스트 한계 극복을 위한 장기 실행 Agent 운영 전략
### 1. 컨텍스트 윈도우의 물리적 제약
 * 수일에 걸친 장기 프로젝트(예: 경쟁사 3곳의 1년간 기술 동향 리서치)는 대화 이력과 중간 결과가 누적되면서 모델의 컨텍스트 윈도우가 물리적으로 가득 차는 한계에 도달합니다.
 * 컨텍스트가 포화되면 중요한 과거 정보가 누락되고 모델의 성능 및 판단 품질이 저하되어 전체 계획의 일관성이 무너집니다.
### 2. 세션 간 맥락 전달을 위한 정보 압축 프로토콜
새 세션을 시작할 때 이전 로그를 그대로 넘기면 비용과 용량 부담이 크므로 정보를 명확히 구분하여 압축·보관해야 합니다.
 * **요약 컨텍스트 (다음 세션 전달용):** 현재 프로젝트 상태 요약, 최종 선택된 핵심 결정 사항 및 이유, 미해결 이슈 및 리스크, 다음 액션 아이템 등 후속 단계에 영향을 주는 중요 항목만 콤팩트하게 남깁니다.
 * **참조 가능한 세부 기록 (별도 보관용):** 장황한 대화 원본 로그, 코드 및 명령어 실행 이력, 실험 결과 데이터셋, 원본 문서 등은 별도의 파일 저장소에 보관하고 필요할 때만 참조 링크 형태로 활용합니다.
 * **실무 권장 병행 전략:** 매 단계 끝나는 시점에는 시스템의 정밀 복원용 **'체크포인트(블랙박스)'**를 자동 저장하고, 세션이나 하루 작업이 끝나는 시점에는 사람이 읽기 좋은 요약본 형태의 **'진행 기록(운행일지)'**을 파일(JSON/Markdown)로 기록하여 새 세션 시작 시 가장 먼저 로드하도록 구현합니다. 코드/문서 작업 시에는 Git Commit 등의 버전 관리 도구와 연동하여 이력을 연계하는 것이 효과적입니다.
## [지침 5] 데이터 보안과 인프라 최적화를 위한 로컬 LLM 서빙
### 1. 로컬 서빙의 필요성
신제품 공정 검증 데이터, 사내 예산안 등 외부 유출이 절대 불가능한 민감 전제 데이터를 외부 클라우드 LLM API로 전송하는 것은 컴플라이언스 위험이 매우 큽니다. 데이터가 사내 네트워크 경계 밖으로 나가지 않도록 사내 GPU 서버 인프라에 직접 독립형 서빙 환경을 구축해야 합니다.
### 2. 서빙 도구 분석: Ollama vs vLLM
 * **Ollama:** 한 줄 명령어로 설치 및 구동이 가능하여 관리가 편리하며, 개발자 개인 PC 환경, 소규모 개발 테스트, 빠른 프로토타입(PoC) 빌드에 최적화되어 있습니다.
 * **vLLM:** 다중 동시 요청 처리에 특화된 고성능 서빙 엔진입니다. 대화 상태 메모리를 거대한 통블록으로 선할당하여 낭비하지 않고 **작은 조각(페이지 단위 메모리 관리)**으로 쪼개어 필요한 만큼만 동적 배분한 뒤 즉시 회수합니다. 자리가 비는 즉시 다음 대기 요청을 투입하므로 GPU 유휴 시간이 최소화되어 실제 서비스 운영 환경에 필수적입니다.
### 3. 단일 GPU 메모리 용량별 권장 모델 매칭 (INT4 정밀도 기준)
*모델 가중치, KV 캐시, 시스템 버퍼 여유 공간을 모두 포함한 대략적인 기준입니다.*
 * **8GB:** 개인 개발 및 가벼운 프롬프트/요약 목적의 **7B급** 모델 탑재 가능 (\Delta), 13B급 이상 지원 불가 (X).
 * **16GB:** **7B급** 여유롭게 가능 (O), **13B급** 조건부 서비스 가능 (\Delta).
 * **24GB:** **7B~13B급** 안정적 구동 (O), RAG 및 일반 대화 처리, **32B급** 조건부 실험 가능 (\Delta).
 * **48GB:** 다중 사용자를 지원하는 운영 환경에서 **32B급** 모델까지 원활하게 구동 가능 (O), **70B급** 조건부 가능 (\Delta).
 * **80GB 이상:** 대규모 동시 처리가 필요한 실제 운영 환경에서 **70B~120B급** 대형 모델까지 안정적으로 자원 할당 가능 (O).
### 4. 양자화(Quantization)와 인프라 보안책
 * **양자화 전략:** 원래 정밀도(FP16) 모델은 메모리를 과도하게 차지하여 감당하기 어렵습니다. 성능 저하를 최소화하면서 파라미터 숫자를 낮은 비트 정밀도로 압축하는 양자화(INT4 등)를 적용하면 지식/추론 성능을 유지하며 **메모리 사용량을 약 40~60% 절감**할 수 있습니다.
 * **보안 중계 장치 배치:** 로컬 LLM 서버(Ollama/vLLM)를 사내망에 그대로 열어두면 인증 제어가 없어 무단 접근 및 장애 위험이 커집니다. 모델 서버는 로컬 내부망으로 감싸고, 앞단에 API Key 인증, 권한 관리, **속도 제한(Rate Limiting)** 및 모니터링 로깅 기능을 수행하는 중계 장치를 필수로 결합해야 합니다.
 * **모니터링 핵심 지표:** 첫 응답 시간(First Token Latency - 임계치 초과 시 사용자 포화 신호), 초당 처리 토큰 수(Tokens Per Second), 대기 중인 요청 수(Queue Length - 서버 증설 판단 기준), GPU 사용률 및 온도를 실시간 대시보드로 관측해야 합니다.
## [지침 6] 운영 배포 전 필수 검토를 위한 거버넌스 체크리스트
안전장치 부재로 인해 프로토타입(PoC) 단계에서 실제 운영 환경(Production) 배포로 이어지지 못하는 격차(통상 50%p 이상 발생)를 극복하기 위해, 배포 전 반드시 다음 4대 통제 요건을 갖추어야 합니다.
### 1. 최소 권한 원칙 기반의 권한 경계 (Authorization Boundary)
 * Agent를 얼마나 신뢰하느냐가 아니라, **"실수했을 때의 피해 범위를 어디까지 격리할 것인가"**를 정의하는 명확한 기준을 세워야 합니다.
 * Agent 시스템별로 명확한 비인간 계정 및 권한 지도를 발급해야 합니다. (예: 리서치/분석 Agent에게 분석용 DB 및 공개 문서 저장소에 대한 **'읽기 전용(SELECT)' 권한만 부여**하고, 운영 DB 접근이나 데이터 수정/삭제/외부 시스템 설정 변경 권한은 철저히 차단).
### 2. 추적성을 위한 구조화된 의사결정 로그 (Decision Logging)
 * Agent가 내린 모든 중요 판단과 인과관계를 사후 검사가 가능한 구조화된 로그로 변조 방지 저장소에 기록해야 합니다.
 * **필수 로그 상세 항목:** 사건 ID, 판단 시간, 구체적 판단 내용, 원인 입력 데이터(사용자 질문/DB 스키마), 선택 근거, 활용 도구 정보, 전후 추적 정보(이전/다음 사건 ID), 처리 소요 시간 및 성공 여부.
### 3. 승인 체크포인트와 강력한 킬 스위치 (Kill Switch)
 * **승인 체크포인트:** 중요한 자원 집행이나 외부 시스템 실행 전에는 무조건 수동으로 사람의 승인을 거치도록 시스템 흐름 상에 Interrupt 관문을 강제합니다.
 * **원클릭 킬 스위치 구축:** 모니터링 중 첫 응답 시간의 비정상적 급증, 대기 큐 폭증, 동일 쿼리 무한 반복 등의 이상 징후 및 오류율 임계치(예: 5% 초과)가 포착되는 즉시, 담당자가 **Agent의 실행을 즉각 강제 중단**할 수 있는 장치를 마련해야 합니다. 작동 시 모든 실행 중지, 신규 요청 차단, 세션 종료 및 리소스 보호 모드 전환이 동시에 수행되며 상태와 로그는 안전하게 보존되어야 합니다.
### 4. 하이브리드 LLM 라우팅 최적화 전략 (실무 판단)
 * **로컬 모델 우선:** 보안 컴플라이언스 요구도가 매우 높고 데이터 민감도가 큰 영역(미공개 신제품 계획 분석, 고유 공정 데이터 처리), 정형화되고 반복적인 업무 중심 구조에 적용하여 비용을 절감합니다.
 * **외부 상용 API 우선:** 데이터가 공개 가능한 일반 정보 영역이거나, 수많은 사용자의 동시 인프라 확장이 필요하거나, 고도의 복잡한 추론 처리가 필요한 최고 성능 요구 단계에 선별 적용합니다.
 * **실무 구현:** 전체 마스터 계획 수립이나 일반 요약 등 민감도가 낮은 상위 노드는 일반 외부 상용 API를 활용하되, 핵심 민감 데이터를 직접 분석·연산하는 특정 실행 노드(Execution Node)만 사내 로컬 서빙 모델을 호출하도록 완벽히 분리·라우팅하는 하이브리드 아키텍처를 적용하는 것이 실무적으로 가장 고도화된 타협점입니다.
