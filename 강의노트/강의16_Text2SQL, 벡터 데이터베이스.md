## 1. ChromaDB 구축

**Router 안에는...
- *Node, Edge, State*로 graph 기반 agent 구조를 만듦
- *예측, 이상 탐지, 검색* 세 그룹으로 Tool을 나눠 문제를 해결

**검색 기능의 성능이 떨어지는 이유들
1) 문서가 새로 추가될 때마다 다시 임베딩 해야 함
2) 임베딩 모델을 데이터와 목적을 고려한 기준 없이 선택
3) 검색 결과를 검증하지 않음
4) 계산이 필요한 숫자 질문도 검색에 의존해버림
검색 성능에는 색인 구조와 임베딩 모델이 중요하다.

**ChromaDB란?**
로컬 우선 벡터 DB. 별도 서버를 구축할 필요가 없으며 초기 개발 및 검증, 사내 데이터 활용 프로젝트에 적합하다.

**Collection이란?**
성격이 같은 문서/같은 목적의 문서들을 모아 담는 하나의 저장 단위.
- 지나치게 세분화된 구조: 여러 collection을 동시에 조회해야 해서 비용이 증가하고 관리 복잡도가 증가한다.
- 지나치게 다 때려넣은 구조: 정확도 저하, 노이즈 증가

#### 문서 인덱싱 파이프라인
1) SOP 문서 불러오기
2) 텍스트 추출(Loader)
3) Chunking & Overlap ; 보통 800~1200자씩 자르고, 100~200자 겹치게 한다.
4) 각 chunk를 벡터로 변환(임베딩 생성)
5) Collection에 벡터와 메타데이터(태깅) 저장
	==메타데이터가 검색 품질을 좌우한다.== 검색 시 이 정보로 범위를 먼저 좁힌 뒤 벡터 유사도 검색을 수행하면 검색 신뢰도 및 속도를 향상시킬 수 있다.

**벡터 "유사도가 높다"는 것은 어떤 의미?**
- 코사인 유사도: 방향이 얼마나 비슷한가? $cos(θ)$가 1에 가까울수록 유사.
-  유클리드 거리: 실제 직선 거리 $||A-B||_2$
-  내적: 방향과 크기를 모두 반영 $|A||B|cos(θ)$

#### 벡터 검색을 빠르게 하는 방법들
문서가 수십만 건으로 늘어나면 질문마다 모든 벡터와 거리를 일일이 계산하는 방식은 너무 느리다. 
- ANN(==Approximate== Nearest Neighbor): 계층 구조를 따라 후보를 좁혀가며(근사) 탐색하는 것이 빠르고 효율적이다.
- 여러 Collection ==동시 조회, 결과 통합==
- 수백 건의 문서를 ==배치 단위로 분할==하여 반복적으로 순차 처리
- Collection의 ==주기적 백업==과 마이그레이션

## 2. Embedding 모델
**주요 선택지**

- bge-m3
	무료|다국어|가성비
	**다국어 병렬 데이터**: 서로 다른 언어라도 의미가 같으면 벡터 공간에서 가깝게 위치하도록 학습된다.
- ko-sroberta
	무료|한국어 특화

**텍스트가 벡터로 바뀌는 과정**
- Bi-encoder 구조: 질문과 문서를 각각 독립적으로 임베딩
	문서는 미리 임베딩 해두고, 검색 시점에 질문만 벡터로 바꿔서 비교

**재변환 비용이라는 숨은 기준**
- 임베딩 모델을 한 번 선택하면, 모델을 바꾸려면 저장된 문서 전체를 처음부터 다시 임베딩 해야 한다. 서로 다른 모델이 만든 벡터는 같은 공간에서 비교할 수 없기 때문이다. 
- 따라서 ==회사 정책을 따르는 것이 좋다.==

**우리 문서에 맞게 더 정밀하게 만들기**
범용 임베딩 모델은 반도체 공정 특유의 표기, 약어, 은어는 잘 모를 수 있다. 
- 사내 데이터를 모아 Fine tuning해버리기
- 하이브리드 검색(BM25+벡터검색): Contextual retrieval로 chunk에 맥락 붙이기
- 도메인 용어 사전을 Query rewriting 단계에 주입

**벤치마크 점수(MTEB:Massive Text Embedding Benchmark)**
- 여러 언어와 다양한 과제에 걸쳐 임베딩 모델의 성능을 종합적으로 측정해 순위를 매긴다.
- 단, 특수한 도메인 문서로 측정한 결과와는 다를 수 있으므로 주의 필요. 항상 1등인 알고리즘은 존재하지 않는다.

**검색 및 추천 성능의 지표 nDCG(normal Discounted Cumulative Gain)**
상위에 노출될수록 관련성 높은 결과를 잘 배치했는지, 결과물이 얼마나 이상적인 순서에 가까운지 정규화하여 0~1사이의 점수로 정량화한다.

## 3. Relevance Checker
**"검색된 문서를 무조건 믿어도 될까?"**
벡터 유사도가 높다해서 무조건 맞는 답변이라는 보장은 없다. 표현이 비슷하나 다른 공정을 다루는 문서가 상위에 올라오는 경우가 흔하다.

**핵심 아이디어** : 검색된 문서를 LLM에 넘기기 전에 먼저 채점 한다.
#### 동작 순서
1) Retriever가 문서를 가져옴(거리 계산)
2) Relevance checker가 각 문서에 관련성 점수를 매기고, 기준치 이상만 통과(판단)
	- LLM 기반 정성적 판단
	- 유사도 기반의 정량적 판단
3) **Cross-encorder**: 순위를 한 번 더 정밀하게 매기기
	Relevance checker가 걸러 넘긴 문서들의 순서를 재배치.
	질문과 문서를 하나의 입력으로 묶어 신경망에 통과시켜 관련도를 직접 계산한다.
	정밀하지만 무겁다.
4) 최종 상위 문서를 LLM에 전달, 답변 생성
	**몇 건을 가져올 것인가?**
	- top-k가 작을 때(적은 문서) : 근거 부족, 답변 누락 위험
	- top-k가 클 때(많은 문서) : 무관한 문서가 많아 핵심 파악 어려움, 비용 증가

**그래프 설계**
- 조건부 Edge에서 "Tool 선택"이 아닌 "문서 관련성 판단"을 적용한다. 관련 문서만 골라서 다음 단계로 넘기고 나머지는 버리는 방식으로 설계.
- ex) 내부 문서 검색이 실패하면 웹 검색 Tool로 자동 전환되게 하고 싶을 때
	Retrieve node -> Relevance check node -> 조건부 Edge
	- 관련 있음 -> Generate node로 연결, 최종 답변 생성
	- 관련 문서 없음 -> 웹 검색 node로 연결, 대체 답변 생성

**웹 검색 API 고르기**
- LLM 활용에 최적화된 검색 API는 결과가 잘 요약, 정제(광고성 정보제거)되어 있어 후처리 부담이 적다.
- 출처의 신뢰도가 제각각이므로 공식 문서나 학술 자료를 우선하고 나머지는 참고용으로 표시하는 것이 안전하다.

**반복 질문을 위한 캐싱** : 비용과 응답 시간의 낭비를 방지
- 자주 묻는 질문과 그 답변, 통과했던 문서 조합을 별도로 저장해두고 비슷한 질문이 또 들어오면 캐시를 먼저 확인하게 한다.
	단, 공정 파라미터처럼 자주 바뀌는 정보는 캐싱 대상에서 제외

## 4. Text2SQL
자연어 질문을 SQL 쿼리로 자동 변환한 뒤 DB에서 실행하고, 결과를 다시 자연어로 설명하는 기술

**벡터 검색** : 개념, 정의, 절차, 원인 등의 설명과 배경 지식이 필요할 때
	ex) "포토 공정에서 focus offset이 뭐야?"
**SQL 검색** : 정형 데이터로부터 정확한 숫자, 집계, 추이 등의 계산과 검증이 필요할 때
	ex) "지난달 ~수율 평균이 얼마야?"

#### Text2SQL(Text to SQL)의 전체 흐름
1) 자연어 질문 입력
2) ==관련 테이블 스키마 정보 제공==
	질문과 관련된 테이블만 선별해서 제공해야 정확도가 향상됨
3) SQL 쿼리 생성(LLM)
4) 쿼리 실행(DB)
5) 자연어로 결과 생성

 **Few-shot 예시로 정확도 높이기**
- 스키마 정보만 주어지면, LLM이 테이블 간 관계를 잘못 이해해 엉뚱한 join을 만들 수도 있다.
- "이런 질문에는 이런 형태의 SQL"과 같은 예시 몇 개를 미리 보여주면, LLM이 그 조직 특유의 테이블 관계와 표기 방식을 더 정확히 따라가게 할 수 있다.

#### Text2SQL의 위험성과 안전 장치
- 실제 기업 DB는 테이블 수가 훨씬 많고, 컬럼명이 축약되어있거나 여러 SQL 방언이 섞여 있다. 데이터가 불완전하기도 하다.
- 의도치 않게 데이터를 변경하는 쿼리(UPDATE, DELETE)를 생성할수도
- 안전장치 : ==읽기 전용 권한==(SELECT만 허용) & 결과 없음 또는 실행 불가와 같은 실패 신호 명시

**운영 DB를 직접 건드리지 않는 법**
- 읽기 전용 복제본(Read Replica)를 활용한다.
- 왜 원본에 연결하면 안되나?
	아무리 안전장치를 갖춘 Text2SQL tool이라도, DB에 예상 밖의 부하를 주면 다른 시스템까지 영향을 받을 수 있다.

**실행 결과를 한 번 더 확인하기**
- SQL이 정상 실행 되었더라도 의도와 다른 결과를 낼 수 있다.
- ==Sanity Check==: 결과값이 상식적인 범위 안에 있는지 점검
	ex) 수율은 0~100%, 비율은 0~1, 건수는 0 이상...

**숫자의 시각화**
추이나 여러 항목 간의 비교를 묻는 질문은 시각 자료가 더 이해하기 좋다.

**실무에 적용: 숫자와 설명을 나눠서 다뤄야 한다.**
벡터 검색은 비슷한 내용을 찾을 뿐이라, 문서 안의 정확한 숫자 데이터는 놓칠 위험이 있다. 따라서 ==수치 부분은 SQL로== 관리하고, ==설명 부분은 검색으로== 관리하는 것이 좋다.

## 5. 실무 적용 가이드
### 1) SOP 문서를 검색 가능하게 만들기
공정별 SOP PDF 수백 건을 매번 사람이 찾아 읽어야 하는 상황.
한 번 인덱싱 해두면 이후에는 바로 검색 가능한 지식 기반이 된다.
1) SOP PDF 로드
2) 인덱싱(절차단위로 분할하는 전처리 -> 임베딩 -> 메타데이터 부여)
3) Collection에 영구 저장

### 2) 검색 결과를 두 단계로 검증하기
"CMP slurry 교체 주기"를 검색했는데 전혀 다른 공정 문서가 섞여 나오는 상황.
1) Retrieve node(벡터 검색). 상위 N개 문서 검색
2) Relevance check node로 문서과 질문의 관련성 판단 
3) 정밀 재정렬 node. 점수로 문서 순서 재정렬 
4) 조건부 edge. 답변을 생성하거나, 통과 문서가 없으면 웹 검색 전환

### 3) 수치는 SQL로 정확하게
"지난달 Lot별 수율 평균이 얼마야?"는 정확한 계산이 필요한 질문이다.
Lot_History과 같은 테이블 스키마와, 자주 쓰이는 조회 예시를 함께 제공해 SQL을 생성하고, 읽기 전용 복제본에서 실행한 뒤 결과를 범위 점검하고 자연어와 표 둘다 반환하는 Tool을 설계한다.
1) 자연어 질문
2) 스키마+예시 참고
3) SQL 생성
4) Read Replica
5) Sanity check
6) 결과 반환
